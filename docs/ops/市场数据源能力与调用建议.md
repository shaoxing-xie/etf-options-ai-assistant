# 市场数据源能力与调用建议（经验沉淀）

> 目标：为后续新增工作流、查询界面、批处理任务提供可复用的数据源选型与调度建议。  
> 性质：操作建议（best practice），非硬约束。  
> 更新日期：2026-04-25

---

## 1. 本次实测结论（先看）

- `global_market_snapshot` 已实现稳定 `overall_quality=ok`，关键瓶颈集中在 `index_futures` 阶段。
- `future.a50` 的主耗时来自 `tool_fetch_a50_data -> ak.futures_global_spot_em` 大表拉取，冷启动可能 40-80s。
- 明确指标与源映射后，采用“固定源直连”优于“多源盲探测”：
  - `future.a50`：缓存 -> 插件 ->（失败时）yfinance 兜底。
- 短 TTL 缓存（A50 30s）可显著降低连续刷新耗时（热路径秒级）。
- `lineage_refs` 分阶段耗时对定位性能问题非常有效，建议保留并持续扩展。

## 3. 性能与限流特征（含规避建议）

### 3.1 yfinance

- 现象：
  - 可能出现 `Too Many Requests / Rate limited`。
  - symbol 维度可用性差异大（同批次可能部分成功、部分空）。
- 规避：
  - 源内节流（最小间隔）+ 受控并发（避免突发）。
  - 快慢两阶段：先短超时无重试，失败腿再走慢路径重试。
  - 结果缓存（短 TTL）减少重复请求。
  - 失败原因写入 `source_raw`，便于 UI/日志追踪。

### 3.2 FMP

- 现象：
  - key 配置错误/未展开时会出现空结果或无数据。
  - 免费计划有调用与带宽上限，超限后成功率显著下降。
- 规避：
  - 必须在配置加载阶段做 key 展开与缺失告警。
  - 把 FMP 放在“全球指数现货”高价值路径，避免无意义重复调用。
  - 统一记录 `source_id/source_raw`，便于额度耗尽排查。

### 3.3 AkShare（尤其 `futures_global_spot_em`）

- 现象：
  - A50 可用，但全表拉取耗时明显，波动可达数十秒。
  - 上游站点风控/网络状态对耗时影响大。
- 规避：
  - 仅对明确需要的标的使用（如 A50），避免把它当通用快速源。
  - 进程内缓存 + 超时熔断，防止拖垮整接口。
  - 任务分层：冷路径允许慢，热路径尽量复用最近值。

## 4. 调用策略建议（面向后续新增功能）

### 4.1 选源原则

- 明确映射优先：指标与数据源关系明确时，直接调主源。
- 同品类优先：期货只用期货源，不用指数替代期货。
- 语义统一：统一输出 `source_id/source_raw/quality_status/degraded_reason`。

### 4.2 调度原则

- 先快后慢：先返回大部分可得结果，再补齐失败腿。
- 分批分源：不同源分组拉取，同源控制最小间隔。
- 缓存分层：高耗时但短时间稳定的腿（A50）建议短 TTL 缓存。
- 可观测优先：每阶段写 `lineage_refs`（成功/失败、耗时、明细）。

### 4.3 失败处理原则

- 配置问题优先显式报错（key 缺失、占位符未展开）。
- 限流问题优先重试退避，不做静默吞错。
- 单腿失败不阻断全局；但要保留精确失败标签用于排障。

## 7. 可复制模板片段（新任务直接套用）

### 7.1 Python：source routing + lineage 模板

```python
import time
from typing import Any


def lineage_event(lineage: list[str], stage: str, ok: bool, elapsed_ms: int, detail: str = "") -> None:
    suffix = f":{detail}" if detail else ""
    lineage.append(f"{stage}|ok={1 if ok else 0}|ms={elapsed_ms}{suffix}")


def run_market_view() -> dict[str, Any]:
    lineage: list[str] = []
    tools: list[str] = []

    # Stage A: cn_index（明确主源）
    t = time.perf_counter()
    cn = fetch_cn_index_main_source()  # e.g. akshare.stock_zh_index_spot_sina
    lineage_event(lineage, "cn_index", cn["ok"], int((time.perf_counter() - t) * 1000), f"items={cn['count']}")
    tools.extend(cn.get("tools", []))

    # Stage B: global spot（统一入口）
    t = time.perf_counter()
    gl = tool_fetch_global_index_spot(index_codes="^HSI,^N225,^GSPC,^IXIC")
    lineage_event(lineage, "global_spot", bool(gl.get("success")), int((time.perf_counter() - t) * 1000), str(gl.get("message") or "ok"))
    tools.append("tool_fetch_global_index_spot")

    # Stage C: futures（先快后慢 + 同品类 fallback）
    t = time.perf_counter()
    fut = build_futures_fast_then_slow()
    ok_cnt = sum(1 for x in fut if x["quality_status"] == "ok")
    lineage_event(lineage, "index_futures", ok_cnt == len(fut), int((time.perf_counter() - t) * 1000), f"ok={ok_cnt}/{len(fut)}")

    return {
        "summary": {"overall_quality": "ok" if ok_cnt == len(fut) else "degraded"},
        "groups": [...],
        "_meta": {
            "source_tools": sorted(set(tools)),
            "lineage_refs": lineage,
        },
    }
```

### 7.4 参数默认建议（起步值）

- 源内节流：`yfinance >= 0.3s`，`global_spot >= 0.8s`
- 快慢两阶段：快路径 `timeout=4s`（无重试），慢路径 `timeout=8s`（少量重试）
- A50 缓存：`ttl=30s`（高频刷新场景）
- 可观测：每个阶段必须写入 `lineage_refs`

### 7.5 工具层验收字段（插件优化后）

- `tool_fetch_global_index_spot`：
  - 必查字段：`source_id/source_raw/source_stage`、`elapsed_ms`、`attempts`、`quality`、`degraded_reason`、`source_route`
  - 失败语义：读取 `failure_code`（如 `CONFIG_MISSING_KEY`、`RATE_LIMITED`）而非仅看 `message`
- `tool_fetch_a50_data`：
  - 实时链路只允许期货源（`futures_global_spot_em`），不可用“指数替代期货”
  - 缓存验收：`cache_hit`、`cache_age_ms`、`source_stage=cache|primary`
  - 降级验收：`quality=degraded` 时必须含 `degraded_reason`（不得伪装实时成功）
