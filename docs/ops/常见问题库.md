## 数据问题

### Q1: 工具返回"Cache miss"或"数据不足"

**问题描述**：调用数据访问工具时返回缓存缺失错误。

**可能原因**：
1. 本地缓存数据不存在
2. 日期范围超出缓存范围
3. 缓存目录/数据源路径配置错误

**解决方案**：
1. **检查本地缓存目录**：
   ```bash
   ls -la ~/etf-options-ai-assistant/data/cache/
   ```

2. **先获取数据**：
  - 使用数据采集工具先获取数据（如 `tool_fetch_etf_historical`）
  - 等待 OpenClaw 数据采集工作流/cron 将数据写入本地缓存

3. **检查缓存文件**：
   ```bash
   ls -la ~/etf-options-ai-assistant/data/cache/
   ```

4. **调整日期范围**：使用更近期的日期范围

**相关工具**：
- `tool_read_etf_daily`
- `tool_read_index_daily`
- `tool_read_option_greeks`

---

### Q2: 数据格式不一致或字段缺失

**问题描述**：返回的数据格式与预期不符，或缺少某些字段。

**可能原因**：
1. 数据源返回格式变化
2. 字段映射错误
3. 数据版本不兼容

**解决方案**：
1. **检查数据源**：确认数据源API是否正常
2. **查看日志**：检查日志中的错误信息
3. **更新插件**：检查是否有插件更新

**相关工具**：
- 所有数据采集工具

---

### 追加：ETF 实时 current_price 取数来源与短缓存

**问题描述**：你发现 `tool_fetch_etf_realtime` 返回的 `current_price` 与预期不一致，或请求频率高导致获取变慢。

**说明**：
1. `tool_fetch_etf_realtime` 的 current_price 取数链路（从上到下）：
   - `mootdx/TDX`（`fetch_stock_realtime`）-> 同花顺 `fund_etf_spot_ths` -> 新浪 `fund_etf_category_sina` -> 新浪 `stock_zh_a_minute`（1分钟收盘兜底）
2. `fund_etf_spot_ths` / `fund_etf_category_sina` 属于“全量列表/快照”类请求：为降低成本，已增加进程内短缓存（默认 TTL 约 45 秒）。

**排查建议**：
- 观察返回中的 `source` 字段，判断当前走的是哪个 provider/兜底链路。
- 若在极短时间内重复调用，同一 provider 的全量列表请求可能命中短缓存。

**相关配置**：`config.yaml -> realtime_full_fetch_cache`

## 计算问题

### Q3: 技术指标计算返回错误

**问题描述**：`tool_calculate_technical_indicators` 返回错误或结果异常。

**可能原因**：
1. 输入数据不足（需要足够的历史数据）
2. 数据格式不正确
3. 计算参数错误

**解决方案**：
1. **检查数据量**：确保有足够的历史数据（至少60个交易日）
2. **检查数据格式**：确保数据包含必要的字段（收盘价等）
3. **调整参数**：增加 `lookback_days` 参数

**示例**：
```python
# 确保有足够的数据
result = tool_calculate_technical_indicators(
    symbol="510300",
    data_type="etf_daily",
    lookback_days=120  # 增加回看天数
)
```

**相关工具**：
- `tool_calculate_technical_indicators`

---

### Q4: 波动率预测耗时过长

**问题描述**：`tool_predict_volatility` 执行时间超过30秒。

**可能原因**：
1. GARCH模型拟合耗时
2. 数据量过大
3. LLM增强调用耗时

**解决方案**：
1. **使用缓存**：结果会自动缓存5分钟，重复调用会更快
2. **减少数据量**：调整 `lookback_days` 参数
3. **禁用LLM增强**：如果不需要，可以禁用LLM增强（需要修改代码）

**相关工具**：
- `tool_predict_volatility`

---
### Q5: 宽基ETF巡检快报里的波动区间仍然偏宽，怎么确认收敛是否生效？

**问题描述**：你观察到快报中的日内 `range_pct` 依然较大，怀疑收敛逻辑没有应用到缓存读取链路。

**建议按以下步骤排查**：

1. **确认你看的缓存是“新口径生成的当天缓存”**  
   快报读取 `data/volatility_ranges/{date}.json`。如果文件是旧版本代码生成的，新收敛不会回写历史缓存。
2. **检查缓存诊断字段**  
   进入 `underlyings -> 510300 -> etf_range`（或兼容旧格式的顶层 `etf_range`），确认是否存在并满足：
   - `range_pct` 已被夹紧到 `config.yaml -> signal_params -> intraday_monitor_510300 -> volatility -> max_intraday_pct` 对应的百分比口径（默认 4.0）
   - `confidence <= 0.6`
   - `clamp_applied` / `clamp_bounds_pct` 字段存在（有则说明收敛逻辑确实运行过）
3. **检查配置是否被改动**  
   若你调整了 `min_intraday_pct/max_intraday_pct`，同一天缓存会按新边界生成。

**相关文档**：`docs/openclaw/宽基ETF巡检快报-日内波动区间收敛说明.md`

## 网络问题

### Q5: 数据采集工具返回网络错误

**问题描述**：数据采集工具返回连接错误或超时。

**可能原因**：
1. 网络连接问题
2. 数据源API限制
3. 防火墙阻止

**解决方案**：
1. **检查网络连接**：
   ```bash
   ping api.example.com
   ```

2. **重试机制**：工具已内置重试机制（最多3次），会自动重试

3. **检查数据源状态**：确认数据源API是否正常

4. **使用备用数据源**：工具会自动切换到备用数据源

**相关工具**：
- `tool_fetch_etf_realtime`
- `tool_fetch_index_realtime`
- `tool_fetch_option_realtime`

### Q6: 飞书/钉钉通知发送失败

**问题描述**：`tool_send_feishu_message` 或 `tool_send_analysis_report` 返回发送失败。

**可能原因**：
1. Webhook URL配置错误
2. 网络连接问题
3. 消息格式错误

**解决方案**：
1. **检查Webhook URL**：
   - 确认URL格式正确
   - 确认URL未过期

2. **检查配置**：
   ```bash
   # 检查本项目 config.yaml
   cat /home/xie/etf-options-ai-assistant/config.yaml | grep -E 'feishu|dingtalk'
   ```

3. **测试连接**：
   ```bash
   curl -X POST "YOUR_WEBHOOK_URL" -H "Content-Type: application/json" -d '{"msg_type":"text","content":{"text":"test"}}'
   ```

**相关工具**：
- `tool_send_feishu_message`
- `tool_send_signal_alert`
- `tool_send_daily_report`
- `tool_send_analysis_report`

## 配置问题

### Q7: 插件/模块导入失败

**问题描述**：工具返回"无法导入模块"错误（例如导入项目的 `src/` 或 `plugins/` 失败）。

**可能原因**：
1. 项目路径配置错误（项目根未被加入 Python 路径）
2. 关键模块文件不存在
3. Python 路径/符号链接问题

**解决方案**：
1. **检查项目根目录**：
   ```bash
   ls -la /home/xie/etf-options-ai-assistant
   ```

2. **检查关键文件**：
   ```bash
   ls -la /home/xie/etf-options-ai-assistant/src/data_collector.py
   ls -la /home/xie/etf-options-ai-assistant/src/data_cache.py
   ```

3. **检查 Python 路径**：确保 `tool_runner.py` 所在项目根会被加入 `sys.path`（主线安装建议直接使用 `install_plugin.sh` 并重启）。

**相关工具**：
- 所有使用项目 `src/` 或本地 `plugins/` 的工具

---

### Q8: OpenClaw插件未注册

**问题描述**：在Dashboard中看不到 `option-trading-assistant` 插件。

**可能原因**：
1. 插件未正确安装
2. Gateway未重启
3. 插件配置文件错误

**解决方案**：
1. **检查插件目录**：
   ```bash
   ls -la ~/.openclaw/extensions/option-trading-assistant/
   ```

2. **重启Gateway**：
   ```bash
   sudo systemctl restart openclaw-gateway
   ```

3. **检查日志**：
   ```bash
   journalctl -u openclaw-gateway -n 50
   ```

4. **重新安装插件**：
   ```bash
   bash install_plugin.sh
   ```

---

### Q9: 工作流执行失败

**问题描述**：工作流创建后无法执行或执行失败。

**可能原因**：
1. 工具名称错误
2. 参数配置错误
3. 依赖关系配置错误

**解决方案**：
1. **检查工具名称**：确保工具名称以 `tool_` 开头
2. **检查参数**：确保参数格式正确
3. **检查依赖**：确保 `depends_on` 中的步骤名称正确

**示例**：
```yaml
steps:
  - name: step1
    tool: tool_check_trading_status
    params: {}
  
  - name: step2
    tool: tool_fetch_etf_realtime
    depends_on: [step1]  # 确保步骤名称正确
    params:
      etf_code: "510300"
```

## 安装问题

### Q10: 环境检查脚本失败

**问题描述**：运行 `check_environment.py` 时某些检查失败。

**解决方案**：


   ```bash
   pip3 install pandas numpy requests akshare
   ```

3. **OpenClaw未安装**：
   - 参考OpenClaw官方文档安装OpenClaw

4. **项目路径/符号链接错误**：
   - 修改脚本中的路径，或创建符号链接

### Q11: 安装脚本执行失败

**问题描述**：运行 `install_plugin.sh` 时出错。

**可能原因**：
1. 权限不足
2. 路径不存在

1. **检查权限**：

3. **手动创建符号链接**：
   ```bash
   cd ~/.openclaw/extensions/option-trading-assistant
   ln -s /home/xie/etf-options-ai-assistant/plugins ./plugins
   ```

## 性能问题

### Q12: 工具执行速度慢

**问题描述**：某些工具执行时间过长。

**可能原因**：
1. 数据量大
2. 网络延迟
3. 计算复杂

**解决方案**：
1. **使用缓存**：工具已内置缓存，重复调用会更快
2. **检查执行时间统计**：
   ```python
   from plugins.utils.performance_monitor import get_execution_stats
   stats = get_execution_stats("tool_name")
   print(f"平均执行时间: {stats['avg_time']:.3f}秒")
   ```
3. **识别慢工具**：
   ```python
   from plugins.utils.performance_monitor import get_slow_tools
   slow_tools = get_slow_tools(threshold=5.0)
   ```

**相关工具**：
- 所有工具（已集成执行时间统计）

---

### Q13: 内存使用过高

**问题描述**：系统内存使用率持续上升。

**可能原因**：
1. 缓存数据过多
2. 内存泄漏
3. 数据量过大

**解决方案**：
1. **清理缓存**：
   ```python
   from plugins.utils.cache import clear_cache
   clear_cache()
   ```

2. **检查资源使用**：
   ```python
   from plugins.utils.performance_monitor import get_resource_monitor
   monitor = get_resource_monitor()
   stats = monitor.get_stats()
   print(f"内存使用: {stats['process']['memory_mb']['avg']:.2f}MB")
   ```

3. **重启Gateway**：定期重启可以释放内存

## 日志和监控问题

### Q14: 日志文件过大

**问题描述**：日志文件占用磁盘空间过大。

**解决方案**：
1. **日志自动轮转**：本项目日志配置已支持自动轮转（最大10MB，保留7天）
2. **清理旧日志**：
   ```bash
   find logs/ -name "*.log" -mtime +7 -delete
   ```
3. **调整日志级别**：设置环境变量 `LOG_LEVEL=WARNING` 减少日志量

---

### Q15: 无法查看执行时间统计

**问题描述**：无法获取工具执行时间统计信息。

**解决方案**：
1. **确保工具已集成执行时间统计**：使用 `@measure_execution_time` 装饰器
2. **查询统计**：
   ```python
   from plugins.utils.performance_monitor import get_execution_stats
   stats = get_execution_stats("tool_name", hours=24)
   ```

3. **检查时间范围**：确保查询的时间范围内有调用记录
