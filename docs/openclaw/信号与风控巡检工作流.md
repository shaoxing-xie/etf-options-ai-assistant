# 信号与风控巡检工作流（工作流A）

**版本**: v1.0  
**更新日期**: 2026-03-03  
**适用环境**: OpenClaw 2026.2.15 + `etf-options-ai-assistant` 新版多 Agent 架构  

---

## 一、背景与目标

- 过去：盘中有多条工作流/定时任务分别负责**盘中监控、信号生成、风险预警**，逻辑分散且多依赖单一 Agent。  
- 现在：通过 `etf_main` 统一调度 `etf_data_collector_agent`、`etf_business_core_agent`、`etf_notification_agent`，形成一条**端到端的“信号+风控巡检”工作流（工作流A）**。  
- 目标：
  - 将**信号生成、集中风控、成本估算、风险审计与 Feishu 通知**整合到一条链路；
  - 在 Tick 数据能力有限（仅 000300.SH / 399006.SZ）情况下，内置**自动降级到分钟线**的策略，并清晰记录原因；
  - 用**可控且可解释的频率**替代此前高频的盘中监控与信号任务（当前配置为关键时段**每 15 分钟**一次，便于兼顾覆盖度与运维成本）。

---

## 二、参与 Agent 与“铁律”

- **入口 Agent**：`etf_main`  
- **业务 Agent**：
  - `etf_data_collector_agent`：数据采集与数据质量报告（含 Tick/分钟降级信息）  
  - `etf_business_core_agent`：依据 `strategy_config.py` 生成候选信号  
  - `etf_notification_agent`：负责飞书通知（含“技术细节摘要”行）
- **统一执行铁律**（在各 Agent 的 SOUL / WORKFLOW 中已写死）：
  1. **先 env**：所有决策前必须通过 `option_trader.py env` 获取环境与数据能力视图（含 Tick 支持、执行模式等）；  
  2. **再 config/strategy_config**：所有策略参数必须从配置与 `strategy_config.py` 读取，禁止在 Prompt 中硬编码；  
  3. **最后 risk_check**：所有潜在下单动作都必须通过 `option_trader.py risk_check` + `risk_engine` 做集中风控，禁止绕过。

## 三、工作流A：信号+风控巡检步骤

### 步骤1：环境视图（env）

- 由 `etf_main` 调用 `option_trader.py env`，统一获取：
  - 当前执行模式：`execution_mode`（`signal_only` / `paper` / `live`）；
  - 不同标的的可用数据能力：是否支持 Tick、分钟线、日线等；
  - 对 Tick 来说，目前仅对：
    - `000300` → `000300.SH`（沪深300指数）
    - `399006` → `399006.SZ`（创业板指）
    开启 Tick 能力，其余指数/ETF 一律视作**仅分钟/日线可用**。

### 步骤2：数据采集与降级策略

- 由 `etf_data_collector_agent` 调用内部工具（封装 `tick_client` 与分钟级数据接口），对所需标的执行：
  - **优先 Tick**（仅 000300/399006）：  
    - 优先 Alltick，其次 iTick；  
    - 出现 `code invalid`、`data=null`、`HTTP 429` 等情况时，自动降级。
  - **自动降级分钟级**：
    - 对非 000300/399006 标的，直接使用分钟数据（不尝试 Tick）；  
    - 对 Tick 失败的情况，从分钟级 K 线中构造近似“最新价/区间”，并在**数据质量报告**中写明：
      - 原计划数据源（Alltick / iTick）、失败原因；
      - 实际使用的数据源（分钟 K 线）与时间范围。

### 步骤3：策略信号生成（基于 strategy_config）

- 由 `etf_business_core_agent` 完成，关键约束：
  - 所有策略（6 大策略，覆盖日内/波段/期权等）必须从 `strategy_config.py` 读取开关、参数与权重；  
  - 不允许在 Prompt/System message 中“临时发明”策略或参数；
  - 生成结果为**候选信号集合**（信号强度、方向、标的、持仓建议等），此时仍处于**信号阶段**，不执行真实下单。

### 步骤4：集中风控与成本估算（risk_check + risk_engine）

- `etf_main` 将每条候选信号转换成标准化的“订单请求”，调用：
  - `option_trader.py risk_check` → `risk_engine.evaluate_order_request`：
    - 校验是否满足全局风险约束（单笔最大损失、日内最大回撤、最大仓位比例等）；  
    - 调用 `_estimate_costs` 输出：
      - `estimated_slippage_pct`（预估滑点百分比）  
      - `estimated_fee_pct`（预估手续费百分比）  
      - `total_cost_pct`（综合成本百分比）
  - 同时通过 `_log_risk_check` 将**精简后的订单与风控结果**写入 JSONL 审计日志：
    - 路径：`logs/risk_checks_YYYYMMDD.jsonl`；  
    - 字段：时间、标的、方向、数量、风控结果、拒绝原因摘要、成本估算等。

### 步骤5：飞书“信号+风控巡检”通知（含技术细节摘要）

- 由 `etf_notification_agent` 汇总上述信息，调用飞书通知插件发送消息，结构一般包含：
  - **标题**：`📊 信号+风控巡检完成（YYYY-MM-DD）`；  
  - **正文主要部分**：
    - 当日当前时点的整体信号概览（多空倾向、主要标的、信号强度分布）；  
    - 风控结论（通过/拒绝的订单数量、主要拒绝原因归类、风险敞口摘要）；  
    - 若处于 `signal_only` 模式，会在文中明确说明“本轮仅生成信号，不执行真实交易”。  
  - **技术细节摘要行（自动附加）**：
    - 例如：  
      - “技术细节摘要：已使用 Alltick Tick 验证沪深300/创业板指突破，其余指数/ETF 统一采用 5 分钟 K 线降级确认；风控已评估单笔/全局风险与预估交易成本，详情见风险审计日志。”  
    - 这行由 `send_feishu_notification.py` 内部逻辑根据通知类型/标题自动拼接，避免每个工作流重复书写。

### 步骤6：执行模式控制（为模拟盘/实盘预留）

- 当前默认：`execution_mode=signal_only`：
  - 工作流A只做**信号+风控巡检 + 通知 + 审计日志**，不触发真实下单；
  - 这样可以在你完全确认链路稳定前，先长期观察行为。  
- 将来切换到 `paper` / `live` 时，建议：
  - 仍然通过工作流A先完成一次信号+风控巡检；  
  - 仅对“通过风控 + 成本可接受”的信号，在额外步骤中调用实际下单接口；  
  - 所有真实交易也必须落风险审计日志，形成一致的审计视图。

## 四、定时任务配置（Cron 视角）

在 `~/.openclaw/cron/jobs.json` 中，工作流A通过 3 条 Cron 任务覆盖关键盘中时段（当前配置约 **19 次/交易日**：早盘 3 次 + 上午 8 次 + 下午 8 次），
所有任务的 `payload.message` 均应继续遵守「研究模式一」(`~/.openclaw/prompts/research.md`) 的统一规范，并保持沿用现有的 Feishu/钉钉关联渠道与群：

| 名称 | Cron任务ID | 执行时间（当前配置） | 说明 |
|------|------------|---------------------|------|
| etf: 信号+风控巡检（早盘） | `etf-signal-risk-inspection-morning` | 工作日 9:15、9:30、9:45 | 开盘早段巡检，结合集合竞价/开盘波动与隔夜信息 |
| etf: 信号+风控巡检（上午） | `etf-signal-risk-inspection-midday` | 工作日 10:00–11:45 每 15 分钟 | 覆盖上午主要波段，监控仓位与风险 |
| etf: 信号+风控巡检（下午） | `etf-signal-risk-inspection-afternoon` | 工作日 13:00–14:45 每 15 分钟 | 覆盖午后与尾盘前风险状态 |

> 注意：原有的 `etf: 交易信号生成`、`etf: 风险监控预警` 以及 4 条“盘中15分钟监控”Cron 任务，已在 `jobs.json` 中设置为 `enabled=false`，以避免和工作流A重复。

### 手动触发示例

```bash
# 触发早盘版巡检
openclaw cron run etf-signal-risk-inspection-morning --timeout 1800000

# 触发上午版巡检（任意一轮）
openclaw cron run etf-signal-risk-inspection-midday --timeout 1800000

# 触发下午版巡检（任意一轮）
openclaw cron run etf-signal-risk-inspection-afternoon --timeout 1800000
```

## 五、与旧工作流/任务的关系

- **被替代的任务**：
  - `etf: 交易信号生成`（原：每 5 分钟生成信号并直接发飞书）；  
  - `etf: 风险监控预警`（原：每 30 分钟单独评估风险并预警）；  
  - `intraday_monitor_5min` 相关的 4 条 Cron（盘中 15 分钟监控）。
- **保留但默认禁用**：
  - 配置与文档仍保留，方便对照、回滚或做 A/B 对比；  
  - `jobs.json` 中这些任务的 `enabled` 字段已设为 `false`。
- **推荐实践**：
  - 盘前/盘后分析仍使用增强版工作流（before/after_close_enhanced）；  
  - 盘中一律以“信号+风控巡检工作流A”为主；  
  - 如需临时高频实验，可以在不影响工作流A的前提下，手动启用某条旧 Cron 进行短期测试，但不建议长期并行运行。
