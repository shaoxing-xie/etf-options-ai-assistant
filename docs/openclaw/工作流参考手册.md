### 5. signal_and_risk_inspection（信号+风控巡检，工作流A）

> 目标：用更低频但更“端到端”的方式，统一完成**信号生成 + 集中风控 + 成本估算 + 飞书巡检报告**，并内置 Tick/分钟级数据降级逻辑和风险审计日志。

**入口 Agent**: `etf_main`  
**涉及 Agent**: `etf_data_collector_agent`，`etf_business_core_agent`，`etf_notification_agent`  
**执行频率（共 7 次/交易日）**:

- 早盘版：9:35（1 次）
- 上午版：10:05、10:35、11:05、11:35（4 次）
- 下午版：13:10、14:10（2 次）

**相关 Cron任务ID**（详见下文 Cron 任务管理表）:

- `etf-signal-risk-inspection-morning`（早盘）
- `etf-signal-risk-inspection-midday`（上午）
- `etf-signal-risk-inspection-afternoon`（下午）

**核心步骤（工作流A）**:

1. **环境视图（env）**  
   - 由 `etf_main` 调用 `option_trader.py env`，通过 `broker_and_data_config` 获取：  
     - 当前执行模式：`execution_mode`（`signal_only` / `paper` / `live`）  
     - 各标的的数据能力视图（是否支持 Tick / 分钟 / 日线等，当前仅对 `000300.SH`、`399006.SZ` 开启 Tick）。  

2. **数据采集（优先 Tick，自动降级分钟级）**  
   - 由 `etf_data_collector_agent` 调用 Tick/分钟级采集工具（内部封装 `tick_client` 与历史分钟线接口）：  
     - 对 `000300.SH`、`399006.SZ`：优先尝试 Alltick/iTick Tick 数据；  
     - 其它标的：直接使用分钟级数据；  
     - 当 Tick 不可用/限流时，自动降级为分钟级数据，并在“数据质量报告”中写明原因（如 `code invalid`、`HTTP 429` 等）。

3. **策略信号生成（strategy_config 单一真源）**  
   - `etf_business_core_agent` 严格从 `strategy_config.py` 读取所有策略配置（6 大策略，禁止在 Prompt 中硬编码参数）；  
   - 基于步骤2的数据，为每个启用策略生成“候选信号”，但此时**严格处于 `signal_only` 阶段，不触发真实下单**。

4. **集中风控与成本估算（risk_check + risk_engine）**  
   - 对每条候选交易请求，通过 `option_trader.py risk_check` 调用 `risk_engine.evaluate_order_request`：  
     - 校验是否满足全局风险约束（单笔/单日损失上限、最大仓位比例等）；  
     - 使用 `_estimate_costs` 估算滑点与费用，输出 `estimated_slippage_pct`、`estimated_fee_pct`、`total_cost_pct`；  
     - 将每次风控请求与结果以 JSONL 形式写入风险审计日志文件，便于事后复盘。

5. **飞书“信号+风控巡检”通知（含技术细节摘要）**  
   - 由 `etf_notification_agent` 调用 Feishu 通知工具，发送一条结构化的“信号+风控巡检”消息：  
     - 概要部分：本轮信号概况、风险评估结论、建议操作（仍以信号/建议为主）；  
     - 明细部分：按策略拆分的信号与风控结果摘要；  
     - **技术细节摘要行**：自动附加一句技术说明，例如：  
       - “已使用 Alltick Tick 验证沪深300/创业板指突破；其它指数/ETF 暂用 5 分钟 K 线降级确认（详见风险审计日志）。”  

6. **执行模式控制（为后续模拟盘/实盘预留）**  
   - 当前默认 `execution_mode=signal_only`，工作流A仅生成信号和风控结论，不自动下单；  
   - 将来当你显式切换到 `paper` 或 `live` 时，可在此基础上挂接“模拟盘/小资金实盘执行”逻辑（仍必须经过 `risk_check`）。

### 手动触发Cron任务

**⚠️ 重要：超时配置说明**

OpenClaw的`cron run`命令默认超时时间为30秒（30000ms），这对于复杂的分析任务（如盘前/盘后分析、盘中监控等）是**远远不够**的。这些任务通常需要：

- 调用多个工具（数据收集、分析、报告生成等）
- 处理大量数据
- 执行LLM推理和分析
- 网络请求和数据传输

**建议所有手动触发的cron任务都添加 `--timeout 1800000` 参数（30分钟）**，以确保任务有足够时间完成。

```bash
# 基本语法（推荐）
openclaw cron run <job-id> --timeout 1800000

# 示例：触发盘后完整分析流程
openclaw cron run 7b401fcc-1f51-4db9-b3e1-74115890a296 --timeout 1800000

# 示例：触发盘前完整分析流程
openclaw cron run 572f20f2-fa1d-4e25-9e0b-fbcccf366790 --timeout 1800000

# 示例：触发盘中监控（任意一个）
openclaw cron run 339222f4-9119-4975-a165-6626451ad236 --timeout 1800000
```

## 最佳实践

### 1. 工作流选择建议

- **盘前分析**: 使用 `before_open_analysis_enhanced`（9:20执行，更完整）
- **盘中监控 / 信号+风控巡检**: 使用 `signal_and_risk_inspection（工作流A）`（当前为关键时段每15分钟运行，端到端完成信号生成与集中风控）
- **盘后分析**: 使用 `after_close_analysis_enhanced`（15:30执行，包含信号生成和记录）

### 2. 手动触发场景

- **测试新工作流**: 在正式启用前手动触发测试
- **紧急分析**: 需要立即获取分析结果时
- **数据修复**: 某个时间段的数据缺失，需要重新分析
- **按需信号**: 使用 `signal_generation_on_demand` 工作流

### 3. 错误处理

所有增强版工作流都包含错误处理机制：
- 数据采集步骤失败时，继续执行后续步骤
- 信号记录失败时，不影响报告发送
- 止损止盈检查失败时，继续执行信号提醒

### 4. 性能优化

- **批量数据获取**: 工作流中已配置批量获取多个ETF和指数数据
- **条件执行**: 信号提醒只在信号强度>=medium时执行
- **缓存利用**: 工具内部会使用缓存机制，减少API调用

### 5. 会话与记忆管理（减少长会话累积）

- **/compact**：在会话中执行，触发当前会话历史压缩，减少上下文膨胀、节省 token。
- **/new**：开启新会话，避免单会话过长；适合阶段性任务（如换日、换主题）后使用。
- **MEMORY.md**：在 workspace 根目录维护 `MEMORY.md`，预写交易偏好、常用标的、阈值与规则；memory flush 与 memory search 会利用该文件与 `memory/` 下日期文件，便于压缩前后保留关键信息。

### 6. Workspace 与大文件

- **不要**把大文件（如整份 Parquet、大 CSV）直接注入 prompt 或让 Agent 扫全目录加载。
- **推荐**：工作流与工具按需读取指定文件或指定列/时间范围；用现有工具（如 `tool_fetch_etf_historical`、文件 read 等）按需取数，避免全量注入。

### 7. 模型与 Fallback 运营优化

- **Fallback 顺序**：在 `openclaw.json` 的 `agents.defaults.model.fallbacks` 中按成本/稳定性调整顺序；可把更便宜或本地模型适当提前。
- **用量与定价**：NVIDIA/DeepSeek 等按实际定价做用量监控；需要降本时可优先让 data_collector、notification 用本地或便宜模型，analysis 保留大模型（当前 Agent 级 model 已分层配置）。
- **单任务覆盖**：对部分 Cron 任务可用 `openclaw cron edit <jobId> --model <provider/model>` 单独指定更便宜模型。


### 问题2: Gateway超时错误

**错误信息**:
```
Error: gateway timeout after 30000ms
```

**原因分析**:
- OpenClaw的`cron run`命令默认超时时间为30秒（30000ms）
- 复杂的工作流任务（盘前/盘后分析、盘中监控等）通常需要：
  - 调用多个工具（数据收集、分析、报告生成）
  - 处理大量数据
  - 执行LLM推理和分析
  - 网络请求和数据传输
- 30秒对于这些任务来说**远远不够**，导致任务在完成前就被超时中断

**解决方案**:
```bash
# 所有手动触发的cron任务都必须添加超时参数
openclaw cron run <job-id> --timeout 1800000

# 1800000ms = 30分钟，足够大多数复杂任务完成
```

**最佳实践**:
- ✅ **总是**在手动触发cron任务时添加 `--timeout 1800000`
- ✅ 对于特别复杂的任务，可以适当增加超时时间
- ❌ **不要**使用默认的30秒超时，这会导致任务失败

### 问题6: Chat 报错 "Access to the current AK has been blocked due to unauthorized requests"

**错误信息**:
```
[API Error] Access to the current AK has been blocked due to unauthorized requests
```

**原因分析**:
- 该提示来自**上游模型 API**（如 SiliconFlow、阿里云 DashScope 等），不是 OpenClaw 本身。
- 请求链路：OpenClaw Chat → IFLOW2API（127.0.0.1:28000）→ 上游厂商 API。
- OpenClaw 里 iflow2api 的 `apiKey` 使用 `dummy-no-key-required-for-local-llama` 是合理的（仅用于访问本地 IFLOW2API）。
- IFLOW2API 在请求上游时使用的是**自己配置的上游 API Key**；该 Key 被厂商封禁或失效时，上游会返回上述错误。

**解决方案**:
1. **确认 IFLOW2API 的上游配置**  
   在运行 IFLOW2API 的环境（同一台机或容器）中查看：
   - 环境变量：如 `SILICONFLOW_API_KEY`、`DASHSCOPE_API_KEY` 或 IFLOW2API 文档中声明的上游 Key 变量名；
   - 或 IFLOW2API 的配置文件（如 `config.yaml`、`.env`）中与 qwen3-coder-plus 等模型对应的 upstream API key。
2. **到厂商控制台处理 Key**  
   - 登录对应厂商（如 [SiliconFlow 控制台](https://cloud.siliconflow.cn/account/ak)）查看该 Key 是否被禁用、限流或过期。  
   - 若 Key 已被封禁，需新建 API Key，并在 IFLOW2API 的配置/环境中替换为**新 Key**。
3. **重启 IFLOW2API**  
   修改上游 Key 后重启 IFLOW2API 服务，再在 OpenClaw Chat 或 Cron 任务中重试。

**快速自测**（在 IFLOW2API 已启动时）:
```bash
curl -s -X POST "http://127.0.0.1:28000/v1/chat/completions" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer dummy-no-key-required-for-local-llama" \
  -d '{"model":"qwen3-coder-plus","messages":[{"role":"user","content":"hi"}],"stream":false}'
```
- 若返回带 `choices` 的 JSON，说明上游 Key 正常。  
- 若返回 500 或 "缺少 choices 数组" / "AK has been blocked"，说明上游 Key 仍无效，需按上面步骤检查并更换 Key。
