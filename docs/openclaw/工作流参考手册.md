### 三条链路对照（防混淆）

以下与仓库根目录 [`research.md`](../../research.md) 第十节 **「三条链路对照」** 一致；调度以本机 **`~/.openclaw/cron/jobs.json`** 为准。勿用 **9:15 早盘采集** 或 **9:28 开盘晨报** 的输出代替 **16:30 每日市场分析报告** 的验收对象。

| 链路 | 典型调度 | Agent | 投递/摘要 | 工具骨架（摘要） |
|------|----------|-------|-----------|------------------|
| 早盘数据采集 | 工作日 ~9:15 | `etf_data_collector_agent` | **飞书**运维摘要；**禁止**钉钉长文、`tool_send_daily_report` | `tool_fetch_index_historical` + `tool_fetch_etf_historical`（priority=high）等，见对应 job payload |
| 开盘行情分析 | 工作日 ~9:28 | `etf_analysis_agent` | 钉钉（[`opening_analysis.yaml`](../../workflows/opening_analysis.yaml)） | Cron 推荐单次 `tool_run_opening_analysis_and_send`；全球指数、北向、要闻、板块热度等见该文件 `structured_message` |
| **每日市场分析报告** | **工作日 ~16:30** | `etf_analysis_agent` | **钉钉**（[`daily_market_report.yaml`](../../workflows/daily_market_report.yaml)） | 推荐：`tool_analyze_after_close_and_send_daily_report`（默认 `report_type=daily_market`，收评式章节）；发送前按 YAML **P0** 合并 sector / 北向 / 关键位 / 政策行业公告 / 大宗 / 外盘 / 波动 / 信号等；可选增强与 [`daily_market_report_web_benchmark.md`](../research/daily_market_report_web_benchmark.md) 对标；关键词弱校验见 `scripts/verify_daily_market_report_keywords.py` |

**每日日报章节对标**：应对齐网上「**交易日收盘复盘 / 当日市场综述**」类公开报告结构，而非早盘机构晨报；清单与 dual evidence 见 [`docs/research/daily_market_report_web_benchmark.md`](../research/daily_market_report_web_benchmark.md)。

**钉钉正文以一份 `report_data` 为唯一展示源**。推荐字段（缺失时对应章节显示占位提示）：

- `market_overview.indices`：来自 `tool_fetch_global_index_spot`（`name`/`code`/`change_pct`）；可选并入 `macro_commodities` 或顶层 `macro_snapshot.items`（原油/黄金/铜）。
- `policy_news`：`tool_fetch_policy_news` 返回的 `data`（含 `items`、`brief_answer`，`source_type=search`）。
- `capital_flow.northbound`：`tool_fetch_northbound_flow` 完整返回；模板须标注 **T-1 / 昨日收盘后口径**。
- `overnight_digest`：`tool_fetch_overnight_futures_digest`（A50/金龙 **定性**，`numeric_unverified=true`）；当 `analysis.overnight_overlay_degraded==true` 时尤其需要。可选 **`tool_conditional_overnight_futures_digest`**：仅在 degraded 为 true 时调用 Tavily，节省配额。
- `macro_snapshot`：`tool_fetch_macro_commodities`（yfinance；无数据或不足 3 类时自动尝试 **Tavily 摘要降级**，条目可能仅有 `digest`、无 `change_pct`）。
- `announcement_digest`：`tool_fetch_announcement_digest` 的 `data`。
- `sector_rotation`：文本摘要或 `tool_sector_heat_score` 结果摘要；静态映射见仓库 `config/hot_sectors.json`。
- `key_levels`：`tool_compute_index_key_levels` 的返回（可包一层 `{"data": ...}`）。
- `analysis` / `analysis_data`：`tool_analyze_before_open`。
- `volatility`、`intraday_range`：波动与日内区间工具。
- `prediction_review`：可选，`tool_get_yesterday_prediction_review`。

详见 `docs/research/opening_morning_brief_roadmap.md` 与 `plugins/notification/send_daily_report.py`（`report_type` 为 `before_open` **或** `opening`/`opening_market` 时共用同一章节顺序；后者标题为「开盘行情分析报告」并带「报告形态：开盘独立完整版」行）。

**文件**: `after_close_analysis_enhanced.yaml`（**仓库内唯一盘后工作流定义**；原 2 步精简版 `after_close_analysis.yaml` 已删除）  
**Cron任务ID**: `7b401fcc-1f51-4db9-b3e1-74115890a296`  
**执行时间**: 工作日 15:30  
**Agent**: analysis_agent  
**支持**: ✅ 定时执行 + ✅ 手动触发

**功能**: 完整的盘后分析流程，包含 7 步（覆盖原精简版中的「分析 + 日报」并增加行情、波动率、信号与记录）

**步骤**:
1. **数据采集** - `data_collector_agent.tool_fetch_etf_realtime`
   - 参数: `etf_code="510300,510050,510500"`
   - 错误处理: 失败继续执行
2. **数据采集** - `data_collector_agent.tool_fetch_index_realtime`
   - 参数: `index_code="000300,000016,000905"`
   - 错误处理: 失败继续执行
3. **盘后分析** - `tool_analyze_after_close`
   - 功能: 分析当日市场情况并预测下一交易日趋势
4. **波动率计算** - `tool_calculate_historical_volatility`（单窗口，兼容既有 YAML）
   - 参数: `symbol="510300"`, `lookback_days=60`（`data_type` 可省略）
   - **可选升级**：一次多窗口/可选锥/可选 IV 时用 `tool_underlying_historical_snapshot`，参数示例：`symbols="510300"`, `include_vol_cone=false`, `include_iv=false`；默认窗口见合并后配置 `historical_snapshot`（来源：`config/domains/analytics.yaml`）
5. **信号生成** - `tool_generate_option_trading_signals`（别名 `tool_generate_signals`）
   - 参数: `underlying="510300"`
   - 输出: signal_id（用于下一步）
6. **信号记录** - `tool_record_signal_effect`
   - 参数: 使用步骤5的signal_id和信号信息
   - 错误处理: 失败继续执行
7. **发送报告** - `notification_agent.tool_send_analysis_report`
   - 参数: `report_type="after_close"`
   - 数据: 使用步骤3和步骤5的结果构建report_data

**手动触发**:
```bash
openclaw cron run 7b401fcc-1f51-4db9-b3e1-74115890a296 --timeout 1800000
```

- **数据修复**: 某个时间段的数据缺失，需要重新分析
- **按需信号**: 使用 `signal_generation_on_demand` 工作流

### 3. 错误处理

所有增强版工作流都包含错误处理机制：
- 数据采集步骤失败时，继续执行后续步骤
- 信号记录失败时，不影响报告发送
- 止损止盈检查失败时，继续执行信号提醒

### 4. 性能优化

- **批量数据获取**: 工作流中已配置批量获取多个ETF和指数数据
- **条件执行**: 信号提醒只在信号强度>=medium时执行
- **缓存利用**: 工具内部会使用缓存机制，减少API调用

### 5. 会话与记忆管理（减少长会话累积）

- **/compact**：在会话中执行，触发当前会话历史压缩，减少上下文膨胀、节省 token。
- **/new**：开启新会话，避免单会话过长；适合阶段性任务（如换日、换主题）后使用。
- **MEMORY.md**：在 workspace 根目录维护 `MEMORY.md`，预写交易偏好、常用标的、阈值与规则；memory flush 与 memory search 会利用该文件与 `memory/` 下日期文件，便于压缩前后保留关键信息。

### 6. Workspace 与大文件

- **不要**把大文件（如整份 Parquet、大 CSV）直接注入 prompt 或让 Agent 扫全目录加载。
- **推荐**：工作流与工具按需读取指定文件或指定列/时间范围；用现有工具（如 `tool_fetch_etf_historical`、文件 read 等）按需取数，避免全量注入。

### 7. 模型与 Fallback 运营优化

- **Fallback 顺序**：在 `openclaw.json` 的 `agents.defaults.model.fallbacks` 中按成本/稳定性调整顺序；可把更便宜或本地模型适当提前。
- **用量与定价**：NVIDIA/DeepSeek 等按实际定价做用量监控；需要降本时可优先让 data_collector、notification 用本地或便宜模型，analysis 保留大模型（当前 Agent 级 model 已分层配置）。
- **单任务覆盖**：对部分 Cron 任务可用 `openclaw cron edit <jobId> --model <provider/model>` 单独指定更便宜模型。

### 问题2: Gateway超时错误

**错误信息**:
```
Error: gateway timeout after 30000ms
```

**原因分析**:
- OpenClaw的`cron run`命令默认超时时间为30秒（30000ms）
- 复杂的工作流任务（盘前/盘后分析、盘中监控等）通常需要：
  - 调用多个工具（数据收集、分析、报告生成）

  - 执行LLM推理和分析
  - 网络请求和数据传输

**解决方案**:
```bash

# 1800000ms = 30分钟，足够大多数复杂任务完成
```

**最佳实践**:
- ✅ **总是**在手动触发cron任务时添加 `--timeout 1800000`
- ✅ 对于特别复杂的任务，可以适当增加超时时间

### 问题6: Chat 报错 "Access to the current AK has been blocked due to unauthorized requests"

**错误信息**:
```
[API Error] Access to the current AK has been blocked due to unauthorized requests
```

**原因分析**:
- 该提示来自**上游模型 API**（如 SiliconFlow、阿里云 DashScope 等），不是 OpenClaw 本身。
- 请求链路：OpenClaw Chat → IFLOW2API（127.0.0.1:28000）→ 上游厂商 API。
- OpenClaw 里 iflow2api 的 `apiKey` 使用 `dummy-no-key-required-for-local-llama` 是合理的（仅用于访问本地 IFLOW2API）。
- IFLOW2API 在请求上游时使用的是**自己配置的上游 API Key**；该 Key 被厂商封禁或失效时，上游会返回上述错误。

**解决方案**:
1. **确认 IFLOW2API 的上游配置**  
   在运行 IFLOW2API 的环境（同一台机或容器）中查看：
   - 环境变量：如 `SILICONFLOW_API_KEY`、`DASHSCOPE_API_KEY` 或 IFLOW2API 文档中声明的上游 Key 变量名；
   - 或 IFLOW2API 的配置文件（如其自身的 `config.yaml`、`.env`）中与 qwen3-coder-plus 等模型对应的 upstream API key。
2. **到厂商控制台处理 Key**  
   - 登录对应厂商（如 [SiliconFlow 控制台](https://cloud.siliconflow.cn/account/ak)）查看该 Key 是否被禁用、限流或过期。  
   - 若 Key 已被封禁，需新建 API Key，并在 IFLOW2API 的配置/环境中替换为**新 Key**。
3. **重启 IFLOW2API**  
   修改上游 Key 后重启 IFLOW2API 服务，再在 OpenClaw Chat 或 Cron 任务中重试。

**快速自测**（在 IFLOW2API 已启动时）:
```bash
curl -s -X POST "http://127.0.0.1:28000/v1/chat/completions" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer dummy-no-key-required-for-local-llama" \
  -d '{"model":"qwen3-coder-plus","messages":[{"role":"user","content":"hi"}],"stream":false}'
```
- 若返回带 `choices` 的 JSON，说明上游 Key 正常。  
- 若返回 500 或 "缺少 choices 数组" / "AK has been blocked"，说明上游 Key 仍无效，需按上面步骤检查并更换 Key。

## 开盘行情分析输出规范（强制）

开盘行情分析报告必须包含以下模块：

1. **市场概况**：主要指数开盘价、涨跌幅、量能偏离度
2. **隔夜外盘**：美股、港股、原油、黄金涨跌及对 A 股影响
3. **板块热点**：领涨/领跌板块及驱动因素
4. **北向资金**：早盘北向流向及信号意义
5. **趋势预测**：基于强度评分的日内趋势判断（偏多/偏空/震荡）
6. **风险提示**：数据延迟、模型假设、外部冲击
7. **数据来源**：数据工具名称、时间戳、版本号
8. **免责声明**：以上内容仅供研究参考，不构成投资建议

**输出结构（固定骨架）**：

```
📊 核心结论（1-3条）
📉 可执行建议/参数方案
⚠️ 风险提示（单独列出）
📂 数据与来源（本次工具与外部站点）
🧭 下一步行动建议（1-3条）
```
