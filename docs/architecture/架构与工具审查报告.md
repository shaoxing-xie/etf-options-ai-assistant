# ETF Stock AI Assistant — 深度代码审查与架构优化分析

> 基于 Cursor Skills 最佳实践（create-skill / create-rule）及 OpenClaw 插件形态的梳理与优化建议，便于后续一起讨论是否落地。

> 当前定位说明：项目默认主线为 A股 / ETF，文中期权相关能力保留为扩展能力说明。

## 一、审查范围与依据

### 1.1 审查范围

- **自定义工具**：OpenClaw 插件 `index.ts` 注册的工具 + Python `tool_runner.py` 调度层 + `plugins/` 下各工具实现。
- **技能/配置**：`config.yaml`、`agents/*.yaml`、工作流、工具参考手册。
- **最佳实践依据**：
  - **create-skill**：描述清晰（WHAT + WHEN）、单一职责、渐进披露、SKILL 主体 <500 行、术语一致。
  - **create-rule**：规则简洁（<50 行）、一规则一关注点、可操作示例。
  - **通用**：单一数据源、DRY、错误处理与降级可观测。

## 二、自定义工具与技能总览

### 2.1 工具分层关系

```
OpenClaw Agent/Workflow
        │
        ▼
  index.ts (Plugin)  ← 50 个 api.registerTool，名称即 tool_xxx
        │
        ▼  execFileAsync(python, [tool_runner.py, toolName, argsJson])
  tool_runner.py
        │
        ├─ ALIASES：旧名 → (新实现名, 注入参数)  兼容 cron/工作流
        ├─ TOOL_MAP：实现名 → ToolSpec(module_path, function_name, params_model?)
        └─ 动态 import + 调用，stdout/stderr 捕获，纯 JSON 输出

        ▼
  plugins/* (merged / data_collection / analysis / notification / sentinel / copilot / backtest)
```

### 2.4 Python 层合并策略（ALIASES + TOOL_MAP）

- **ALIASES**：保留旧工具名，映射到合并后的实现并注入参数（如 `tool_fetch_index_realtime` → `tool_fetch_index_data` + `data_type: "realtime"`），兼容现有 cron/工作流。
- **TOOL_MAP**：以「实现名」为 key，指向 `merged/*` 或单一模块的 `module_path` + `function_name`。
- 合并后的实现包括：  
  `tool_fetch_index_data`、`tool_fetch_etf_data`、`tool_fetch_etf_minute_direct`、`tool_fetch_option_data`、  
  `tool_read_market_data`、`tool_send_feishu_notification`、`tool_analyze_market`、`tool_volatility`、  
  `tool_position_limit`、`tool_stop_loss_take_profit`、`tool_strategy_analytics`、`tool_strategy_weights`。

这一层设计是合理的：对外保持 50 个语义清晰的工具名，对内收敛到较少实现，便于维护。

## 三、架构与实现上的问题与优化方向

### 3.1 单一数据源：工具清单与参数 Schema

- **现状**：工具名称、参数、说明在 **index.ts** 与 **tool_runner.py / 各 plugin** 两处存在，文档《工具参考手册》再描述一遍。
- **风险**：增删改工具或参数时容易漏改一侧，文档与实现不一致（例如文档已写 `tool_detect_market_regime` 等，但 index 未注册）。
- **建议**（符合 create-skill 的“单一事实源”思路）：
  - 维护一份 **工具清单 + 参数 schema**（如 YAML/JSON 或 TypeScript 类型），作为唯一来源。
  - index.ts 的注册逻辑由该清单 **生成**（或通过脚本生成 index 片段），避免手写 50 段重复结构。
  - 文档中的工具列表/参数表可从同一份清单生成或引用，减少漂移。

### 3.2 index.ts 重复与脚本路径

- **现状**：50 个工具均为同一模式：`api.registerTool({ name, label, description, parameters }, { name })` + `callPythonTool(scriptPath, toolName, params)`；`scriptPath` 写死为 `process.env.HOME + "/etf-options-ai-assistant/tool_runner.py"`。
- **问题**：  
  - 大量重复代码，修改成本高。  
  - 路径与部署方式耦合，无法通过插件配置或环境变量覆盖。
- **建议**：
  - 用 **工具清单** 驱动循环注册，仅对少数特殊工具（如参数或行为差异大）单独写一块。
  - 脚本路径从 `plugin configSchema` 或环境变量读取，例如 `optionTradingAssistantScriptPath` 或 `OPTION_TRADING_ASSISTANT_SCRIPT_PATH`，默认再 fallback 到当前路径。

### 3.3 参数一致性与特殊分支

- **现状**：  
  - `tool_predict_intraday_range` 在 tool_runner 里把 `underlying` 转成 `symbol`，属于参数名不一致的特殊处理。  
  - 其它工具多为「index 传什么，Python 就收什么」，但 schema 仍在两处维护。
- **建议**：
  - 在工具清单/schema 中统一约定「对外参数名」与「Python 函数参数名」，要么在清单中写清映射，要么在 tool_runner 的通用层做一次「schema 驱动」的转换，避免散落 if/else。
  - 若保留 `underlying` → `symbol`，建议在 TOOL_MAP 或 ToolSpec 里用显式字段（如 `param_mapping`）表达，而不是写死在 main() 里。

### 3.4 错误处理与可观测性

- **现状**：tool_runner 已做 stdout/stderr 捕获、JSON 输出、异常时带 `error`/`captured_stdout`/`captured_stderr`，callPythonTool 有 timeout、maxBuffer。
- **可加强点**：
  - 为部分耗时或关键工具增加 **结构化日志**（如 tool_name、params 摘要、duration、success/fail），便于与 OpenClaw 日志关联。
  - 错误码：当前部分 Python 工具返回 `error_code`（如 `VALIDATION_ERROR`），可在一份「错误码表」中统一定义，便于 Agent/工作流分支处理。

### 3.5 配置与敏感信息

- **现状**：config.yaml 中大量使用 `${FEISHU_*}` 等环境变量占位，符合“不把密钥写进仓库”的实践。
- **建议**：在文档或 README 中明确列出「运行 OpenClaw 插件所需的环境变量」（例如 `FEISHU_WEBHOOK_URL`、`OPTION_TRADING_ASSISTANT_PYTHON`、可选 `OPTION_TRADING_ASSISTANT_SCRIPT_PATH`），与 create-skill 中“所需依赖与配置写清楚”一致。

## 四、与 Skills 最佳实践的对照

### 4.1 create-skill 要点

- **描述清晰（WHAT + WHEN）**：每个工具的 `description` 建议同时说明「做什么」和「何时用」（例如盘前/盘中/研究场景），便于 Agent 选工具。
- **单一职责**：当前按「数据/分析/风控/通知」划分合理；研究类工具（market_regime、etf_rotation、strategy_research）若暴露到 OpenClaw，建议在描述中注明「研究级」或「非实时交易」。
- **渐进披露**：工具参考手册已很长，可考虑拆成「常用工具速查」+「按场景的详细说明」+「研究/涨停回马枪专题」，主文档 <500 行，其余用链接或子文档。
- **术语一致**：全项目统一「标的/ETF/指数/合约」等用词，避免同一概念多名称。

### 4.2 create-rule 要点

- 若为项目写 Cursor 规则（如「改 index 时要同步工具清单」），建议单条规则 <50 行、一个关注点、带 1～2 个具体修改示例。

### 4.3 技能/插件边界

- **A股+宽基 ETF 主链路**：已完整暴露，适合作为默认技能集；期权能力建议按需启用。
- **研究/选股/涨停回马枪**：能力在 Python 已具备，是否全部暴露到 OpenClaw 可策略性选择：  
  - 若希望 Agent/工作流做「策略研究」「选股」「回测」，可把 `tool_detect_market_regime`、`tool_etf_rotation_research`、`tool_strategy_research`、`tool_fetch_stock_financials`、`tool_quantitative_screening` 以及涨停/回测相关工具选一批注册；  
  - 若仅给人类在命令行/cron 用，可保持现状，但在文档中注明「仅限 Python/脚本调用，OpenClaw 未注册」。

## 五、可落地的优化清单（供讨论）

| 优先级 | 项 | 说明 |
|--------|----|------|
| P0 | 工具清单单一数据源 | 新增一份工具清单（含 name、description、parameters schema），用其生成 index 注册代码并驱动文档，减少三处不一致。 |

## 六、实施状态（已落地）

- **P0 工具清单**：已新增 `config/tools_manifest.yaml`（67 个工具，含参数 schema），并可通过 `python scripts/generate_tools_json.py` 生成 `config/tools_manifest.json`。
- **P0 研究/选股/涨停/回测开放**：上述工具已在 manifest 中定义，并由插件从 manifest 统一注册，全部对 OpenClaw 开放。
- **P1 清单驱动 index**：`index.ts` 已改为从 manifest 加载并循环注册，脚本路径支持插件配置 `scriptPath` 或环境变量 `OPTION_TRADING_ASSISTANT_SCRIPT_PATH`。
- **P2 参数映射**：`tool_runner.py` 中 `ToolSpec` 已增加 `param_mapping`，`tool_predict_intraday_range` 的 `underlying`→`symbol` 已改为显式配置，并移除硬编码分支。
- **P2 文档渐进披露**：已拆分为《工具参考手册》概述 + [工具参考手册-速查](reference/工具参考手册-速查.md) + [工具参考手册-场景](reference/工具参考手册-场景.md) + [工具参考手册-研究涨停回测](reference/工具参考手册-研究涨停回测.md) + [错误码说明](reference/错误码说明.md)。

## 七、小结

- **工具层**：Python 侧通过 ALIASES + TOOL_MAP 做了合并与兼容，设计清晰；OpenClaw 侧 50 个工具与文档基本对齐，但存在「文档已描述、index 未注册」的研究/选股/涨停/回测工具。
- **架构**：主要可优化点在于「工具清单 + 参数 schema」单一数据源、index 由清单驱动、脚本路径与参数映射可配置可观测，以及文档的渐进披露与术语一致。
- **Skills 实践**：按 create-skill/create-rule 的简洁、单一职责、渐进披露和一致术语做一轮收口即可。

**实施完成后**：修改工具或参数时只需编辑 `config/tools_manifest.yaml`，执行 `python scripts/generate_tools_json.py`，重载或重启 OpenClaw 插件即可生效。
